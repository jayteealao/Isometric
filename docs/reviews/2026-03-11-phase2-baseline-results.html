5. Optimization Priority Ranking

Ordered by end-to-end impact, not implementation convenience.

1. Hit-test cost is still the first thing to attack

Continuous interaction averages 38.3472ms hit-test time across the matrix. That is already larger than the entire frame budget by N=100 and becomes dominant at N=200.

  • s200_m50_continuous: hit-test 106.7288ms, frame 225.5450ms
  • s200_m10_continuous: hit-test 102.0609ms, frame 212.4941ms
  • s100_m50_continuous: hit-test 37.5431ms, frame 81.9583ms
  • s100_m10_continuous: hit-test 35.7603ms, frame 77.5381ms

Primary branches to validate first: spatial-index and broad-phase-sort.

2. Prepare cost is the next hard ceiling

Non-interactive rows still average 36.6401ms prepare time. That means scene preparation remains expensive even before interaction is introduced.

  • s200_m50_none: prepare 102.8262ms, frame 113.6884ms
  • s200_m10_none: prepare 97.1350ms, frame 105.1487ms
  • s100_m50_none: prepare 37.1022ms, frame 42.5213ms
  • s100_m10_none: prepare 35.4177ms, frame 39.8387ms

Prepared-scene-cache remains the most likely branch to improve steady-state rendering.

3. Draw is smaller, but allocations and GC are not

Average draw time is only 1.2253ms, but average allocation is 13.19MB and average GC count is 11364 across the matrix. That makes memory churn part of the story even when draw time is not the top-line bottleneck.

  • s200_m50_continuous: draw 2.5526ms, frame 225.5450ms
  • s200_m10_continuous: allocated 25.00MB
  • s200_m50_continuous: 42338 GC invocations
  • Occasional interaction average hit-test cost is only 0.6724ms

Path-caching and native-canvas need to show either frame-time wins or lower allocation churn to justify priority.

10. Reading Guidance

How to interpret noisy rows and decide what matters.

What To Trust Most

  • Use avgFrameMs for branch-level user-visible impact.
  • Use avgPrepareMs and avgHitTestMs to identify the subsystem creating that frame cost.
  • Use p95 and p99 to judge tail behavior; several heavy rows have much worse tails than their averages suggest.
  • Use allocatedMB and gcInvocations to spot branches that simply move work into allocation churn.

What This Baseline Is For

  • This is the new comparison source for rebased Phase 3 branches.
  • It reflects the updated harness, including spatial-index correctness fixes, test-lane fixes, generalized runner flags, and keep-screen-on.
  • It should replace the older Phase 2 report when judging whether a branch is a real win.
  • For future branch runs on this machine, use Git Bash for the runner rather than WSL bash.
