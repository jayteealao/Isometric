Performance Engineering on an Android Isometric Engine — Part 2: Coplanar Faces and Topological Sort

Series overview. This is the second instalment of a record of performance and correctness work on the Isometric Kotlin library. Part 1 covered the benchmark harness, the prepared-scene cache, broad-phase sort, spatial index, path caching, and the combined run. Part 2 addresses a correctness bug in the depth sorter — coplanar face ordering — and replaces the topological sort implementation with Kahn's algorithm. The benchmark run that follows will measure the impact.

The problem: z-fighting on tile grids

The TileGrid composable lays out a grid of Prism shapes at integer positions. A 6x6 grid of default Prism() (1x1x1 cubes) places prisms at (0,0,0), (1,0,0), (2,0,0), ..., (5,5,0). Adjacent prisms share a boundary plane: the right face of the prism at (0,0,0) is coplanar with the left face of the prism at (1,0,0). They occupy the exact same plane at x=1.

Visually, this produces dark triangular gaps and missing faces — the classic "z-fighting" artifact. But unlike GPU z-fighting (which comes from depth-buffer precision limits), this z-fighting comes from the depth sorter's inability to order coplanar faces.

The workaround was to shrink each prism:

Shape(
    geometry = Prism(width = 0.9, depth = 0.9, height = 0.3),
    ...
)

This left a 0.1-unit gap between adjacent tiles, ensuring no shared boundary planes. The faces were no longer coplanar, so the depth sorter could order them normally. But the visual result was a grid with visible seams — not a rendering bug per se, but a design compromise forced by a limitation in the sort.

Why coplanar faces break the sort

How closerThan works

The depth comparison between two faces uses Path.closerThan(pathA, observer). This function answers: "is this path closer to the observer than pathA?"

The implementation works by counting vertices of each path that are on the observer's side of the other path's plane. For two paths A and B:

  1. Count how many vertices of B are on the same side as the observer relative to A's plane.
  2. Count how many vertices of A are on the same side as the observer relative to B's plane.
  3. Return the difference.

A positive result means "this path is closer" (draw it later). A negative result means "this path is farther" (draw it first). A zero result means neither path is definitively closer — the faces are coplanar, or the geometry is ambiguous.

The zero case

When closerThan returns 0, the old code did nothing:

val cmpPath = itemA.item.path.closerThan(itemB.item.path, observer)
if (cmpPath < 0) {
    drawBefore[i].add(j)
} else if (cmpPath > 0) {
    drawBefore[j].add(i)
}
// cmpPath == 0 → no edge added

No dependency edge means the topological sort has no constraint between these two faces. They can be emitted in any order. For non-overlapping faces, this is fine — if two faces don't overlap on screen, their draw order doesn't matter. But checkDepthDependency is only called for faces that do overlap on screen (that's the intersection check at the top of the function). Two coplanar adjacent faces share a boundary edge, which means their projected polygons do overlap at that edge. The sort needs to pick an order, and the old code picked none.

The consequence is frame-dependent: the order depends on whatever the topological sort's tie-breaking behaviour happens to be, which depends on the order items appear in the dependency graph, which depends on the broad-phase grid iteration order, which depends on hash map bucket ordering. On some frames, the order is correct. On others, a closer face is drawn first and a farther face paints over it, producing a dark gap where the closer face's colour should be visible.

The fix: coplanar tie-breaking

Requirements for the tie-breaker

A valid tie-breaking function for a dependency graph must satisfy two properties:

  1. Antisymmetric: tieBreaker(A, B) == -tieBreaker(B, A). If A should be drawn before B, then B should not also be drawn before A.
  2. Transitive: If A is before B and B is before C, then A must be before C. Transitivity prevents the tie-breaker from introducing cycles into the dependency graph, which would defeat the topological sort.

A simple comparator on a scalar value satisfies both properties. The depth value (x + y - 2z, averaged over vertices) is the natural candidate — it is the same metric the engine already uses for initial ordering, and it is a single scalar comparison.

The three-level comparison

private fun coplanarTieBreaker(pathA: Path, pathB: Path): Int {
    // Level 1: average depth
    val depthDiff = pathA.depth - pathB.depth
    if (depthDiff > 1e-9) return -1   // A farther → A before B
    if (depthDiff < -1e-9) return 1

    // Level 2: first vertex, lexicographic (x, y, z)
    val a0 = pathA.points[0]
    val b0 = pathB.points[0]
    val dx = a0.x - b0.x
    if (dx > 1e-9) return -1
    if (dx < -1e-9) return 1
    val dy = a0.y - b0.y
    if (dy > 1e-9) return -1
    if (dy < -1e-9) return 1
    val dz = a0.z - b0.z
    if (dz > 1e-9) return -1
    if (dz < -1e-9) return 1

    // Level 3: truly identical paths — either order is visually identical
    return 1
}

Level 1 resolves the vast majority of coplanar face pairs. Two faces on the same boundary plane (e.g., the right face of prism (0,0,0) and the left face of prism (1,0,0)) are coplanar but have different centroids, so their depth values differ. The face with higher depth is farther from the viewer and should be drawn first.

Level 2 handles the degenerate case where two coplanar faces have the same average depth. This can happen when two faces share a centroid but differ in vertex arrangement — geometrically unusual but not impossible in pathological scenes. Lexicographic comparison on the first vertex provides a stable tie-break.

Level 3 is the absolute fallback: if two paths are vertex-identical (same points in the same order), they produce the same visual output in either order. Returning a constant ensures the function always picks a side, which guarantees a dependency edge is always added. This prevents the "no edge, arbitrary order" problem that caused the z-fighting.

The epsilon choice

The epsilon 1e-9 matches the threshold used in Path.countCloserThan():

if (observerPosition * pPosition >= 0.000000001) {
    result++
}

Using the same epsilon across the pipeline is important. If closerThan considers two faces "equal" (returns 0) using one epsilon, and the tie-breaker then uses a tighter epsilon that considers them "different" in an inconsistent direction, the overall ordering could be wrong. Consistent thresholds ensure that the boundary between "definitively ordered" and "needs tie-breaking" is the same everywhere.

Replacing the topological sort

Why the old sort was O(V2)

The old topological sort used a repeated-scan approach:

while (drawThisTurn) {
    drawThisTurn = false
    for (i in 0 until length) {
        if (!drawn[i]) {
            val canDraw = drawBefore[i].all { drawn[it] }
            if (canDraw) {
                sortedItems.add(items[i])
                drawn[i] = true
                drawThisTurn = true
            }
        }
    }
}

On each round, every undrawn item is checked: "have all my dependencies been drawn?" If yes, draw it. Repeat until no progress is made. In the worst case — a linear chain where item 0 depends on item 1, which depends on item 2, etc. — each round draws exactly one item, and each round scans all remaining items. This is V rounds times V items: O(V2).

For the benchmark scenario (200 items, 3 faces each, 600 total), the sort is over 600 items. The old approach could require up to 600 rounds of 600 checks = 360,000 operations in the worst case. In practice, the dependency graph is wide (many items have zero or few dependencies), so the average case is better than worst case. But the improvement is hard to predict, and the worst case is reachable for certain scene configurations.

Kahn's algorithm: O(V+E)

Kahn's algorithm maintains an explicit in-degree count for each node and a queue of nodes with zero in-degree. Instead of scanning all nodes to find ready ones, it processes only nodes that are known to be ready:

// Build in-degree and reverse adjacency
val inDegree = IntArray(length) { drawBefore[it].size }
val dependents = List(length) { mutableListOf<Int>() }
for (i in 0 until length) {
    for (j in drawBefore[i]) {
        dependents[j].add(i)
    }
}

// Seed queue
val queue = ArrayDeque<Int>()
for (i in 0 until length) {
    if (inDegree[i] == 0) queue.addLast(i)
}

// Process
while (queue.isNotEmpty()) {
    val node = queue.removeFirst()
    sortedItems.add(items[node])
    for (dep in dependents[node]) {
        inDegree[dep]--
        if (inDegree[dep] == 0) queue.addLast(dep)
    }
}

Each node is enqueued exactly once and dequeued exactly once. Each edge is traversed exactly once (when decrementing in-degrees). Total work: O(V + E), where E is the number of dependency edges.

The data structures

inDegree: IntArray(length) — the number of unsatisfied dependencies for each item. Initialised from drawBefore[i].size. When this reaches zero, the item is ready to draw.

dependents: List<MutableList<Int>> — reverse adjacency. If item i must be drawn before item j (i.e., j in drawBefore[i]), then dependents[j] contains i. This reversal is necessary because Kahn's algorithm processes nodes in topological order: when a node is drawn, it needs to know which other nodes were waiting on it.

queue: ArrayDeque<Int> — standard BFS queue. Seeded with all zero-in-degree nodes in index order. The "in index order" part is important for determinism: if two items have no dependencies, the one with the lower index is drawn first. This matches the old sort's behaviour (it iterated 0 until length on each round) and ensures that scenes with no depth ambiguity produce identical output.

Cycle handling

The old sort handled cycles implicitly: after the while loop exhausted, any undrawn items (those trapped in a dependency cycle where no item has all dependencies satisfied) were appended in index order. Kahn's algorithm handles cycles the same way — items with inDegree[i] > 0 after the queue is exhausted are part of a cycle and are appended as a fallback:

for (i in 0 until length) {
    if (inDegree[i] > 0) {
        sortedItems.add(items[i])
    }
}

Cycles in the dependency graph represent genuinely ambiguous ordering — three faces where A occludes B, B occludes C, and C occludes A. No correct order exists. The fallback ensures all items appear in the output (no faces are lost), which is the same guarantee the old sort provided.

Why these changes interact

The coplanar tie-breaker and the Kahn's algorithm replacement are independent fixes that address different symptoms of the same root cause, but they reinforce each other in an important way.

Without the tie-breaker, Kahn's algorithm would still produce arbitrary order for coplanar faces. The tie-breaker adds the missing edges. Kahn's algorithm consumes those edges and produces the correct order — faster than the old sort would have.

Without Kahn's, the tie-breaker would still fix coplanar ordering — but at O(V2) cost. The tie-breaker adds more edges to the dependency graph (every coplanar pair now has an edge), which means more dependencies per item, which means more rounds in the old sort before items become drawable. Kahn's algorithm handles the denser graph in the same O(V+E) time.

The density increase from the tie-breaker is bounded. In a tile grid, each tile has at most 4 adjacent tiles (the cardinal neighbours). Each shared boundary contributes at most 2 coplanar face pairs (left/right or top/bottom). For a 10x10 grid with 100 tiles and 600 faces, the tie-breaker adds roughly 360 extra edges — a modest increase over the existing edge count from non-coplanar depth comparisons. Kahn's algorithm handles this without measurable overhead.

The test strategy

What the tests verify

Four tests cover the fix:

Deterministic order for coplanar adjacent prisms. Two prisms at (0,0,0) and (1,0,0) share a boundary at x=1. The test projects the scene twice with independent engine instances and asserts identical command order. Before the fix, this test would fail intermittently (order depended on hash map iteration, which is JVM-version-dependent). After the fix, the tie-breaker produces the same edge regardless of iteration order, so the topological sort is deterministic.

Face count and uniqueness for a 3x3 grid. Nine 1x1x1 prisms, no shrinking. Asserts 54 faces (9 prisms times 6 faces with NoCulling) and all unique command IDs. This is the direct regression test for the z-fighting bug: if any face were lost or duplicated due to a sort error, the count or uniqueness assertion would fail.

Broad-phase parity for coplanar grids. The coplanar grid scenario with and without enableBroadPhaseSort. Asserts identical command ID order. This verifies that the broad-phase grid does not miss any coplanar pairs — if the grid bucketed two coplanar faces into different cells (because their bounding boxes barely miss the cell boundary), the broad-phase run would lack the tie-breaker edge and produce a different order.

Cycle fallback completeness. Three overlapping prisms at staggered heights. Even if the geometry creates circular dependencies, all 18 faces (3 prisms times 6) must appear in the output. This tests Kahn's cycle fallback — the inDegree[i] > 0 append loop must not lose items.

What the tests do not verify

The tests do not verify visual correctness — that the draw order produces a visually correct image. Visual correctness is a property of the depth comparison function (closerThan), not the sort algorithm. The sort is correct if it respects all dependency edges, which the deterministic-order test verifies indirectly (same edges in, same order out). Visual correctness requires a screenshot comparison or manual inspection, which the TileGrid sample provides.

Reverting the workaround

With the tie-breaker in place, the TileGrid example reverts from:

geometry = Prism(width = 0.9, depth = 0.9, height = 0.3)

to:

geometry = Prism()

The default Prism() produces a 1x1x1 cube. Adjacent cubes share boundary planes. The tie-breaker ensures deterministic ordering of those coplanar faces. The visual result is a seamless tile grid with no gaps between tiles.

This revert is the clearest signal that the fix works: the workaround was the only thing preventing coplanar boundaries, and removing it exercises the exact code path the fix addresses. If the depth sorter still had the coplanar bug, the TileGrid tab would show the original z-fighting artifacts.

What to measure next

The fix changes two things that affect performance:

  1. More dependency edges. The tie-breaker adds an edge for every coplanar face pair that overlaps on screen. This increases the E in O(V+E). For tile grids, the increase is proportional to the grid perimeter (interior faces are culled in production; boundary faces are the ones that overlap neighbours). The question is whether this additional edge count measurably affects frame time.
  2. Different sort algorithm. Kahn's O(V+E) replaces the old O(V2) worst-case scan. For scenes where the dependency graph is deep (long chains), Kahn's should be strictly faster. For scenes where the graph is wide (few dependencies per item), the old scan's average case was already near O(V), and Kahn's adds the overhead of building the reverse adjacency and in-degree arrays. The question is whether the overhead matters for small scenes (N=10) and whether the improvement matters for large scenes (N=200).

The benchmark run should cover the existing 24-scenario matrix with the new code to establish whether the changes are performance-neutral (the goal — correctness fix, not optimisation) or whether they shift the frame-time distribution.
