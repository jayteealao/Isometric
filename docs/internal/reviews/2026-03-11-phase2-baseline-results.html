Where the frame-time gains come from

The renderer is prepare-bound, not draw-bound. Branches that reduce prepare time produce frame-time wins; branches that only reduce draw time do not. This is the single most important finding across all five branches.

Frame-time gains are prepare-driven

The baseline spends 36.6ms in prepare vs 1.2ms in draw on average. Prepare is 30× larger than draw. This means:

  • Broad-phase sort wins because it cuts prepare by 50–69% at scale — the largest lever in the system.
  • Prepared-scene-cache wins because it eliminates redundant prepare work across frames.
  • Path caching and native canvas both cut draw 46–66%, but draw is only 1.2ms — cutting it in half saves 0.6ms, invisible next to a 30ms prepare improvement.

The conclusion is clear: optimize prepare first, draw later. Draw-focused branches only become impactful after prepare has been reduced enough to change the cost balance.

Scaling behavior at N=200

All branches except native canvas improve frame time at N=200. The improvement gradient is steep:

  • BPS: −73.5ms avg at N=200 (dominant)
  • PSC: −52.1ms avg at N=200
  • Path caching: −20.9ms avg at N=200
  • Spatial index: −5.7ms avg at N=200 (partial)
  • Native canvas: −5.7ms avg at N=200

At N=10, all branches converge to the vsync floor (8.33ms) — there is no optimization opportunity below the display refresh boundary.

Memory allocation across all branches

Average allocation delta (branch minus baseline) for each branch, broken down by scene size. Positive means more memory used. None of the branches show systematic allocation inflation — the dominant cost driver is algorithmic, not memory.

Memory allocation wins

No branch introduces a dramatic memory penalty:

  • Broad-phase sort: −0.3MB average — reduces allocation slightly due to fewer sort operations per frame.
  • Prepared scene cache: −0.8MB average — cache reuse eliminates repeated allocation.
  • Path caching: +1.1MB average — modest cost from cached Path objects. Acceptable tradeoff for draw savings.
  • Spatial index: +0.2MB average — essentially neutral.
  • Native canvas: +0.4MB average — slight interop cost.

The key insight: prepare-focused optimizations tend to reduce allocations because they eliminate redundant object creation. Draw-focused optimizations tend to add modest allocation from caching.

GC reduction

Garbage collection count is dominated by frame count and allocation churn, not optimization choice:

  • Broad-phase sort: −32% GC count average — the largest reduction. Fewer prepare iterations means fewer temporary allocations and fewer GC triggers.
  • Prepared scene cache: −14% GC count average — cache reuse reduces churn.
  • Path caching: +3% — slight increase from cache management overhead.
  • Spatial index: +5% — minor increase from index maintenance.
  • Native canvas: −1% — essentially neutral.

BPS achieves the largest GC reduction because it directly reduces the number of iterations in the prepare loop, which is the main source of temporary object allocation.

Recommended merge order

1. Broad-phase sort — highest impact (−30.7ms), zero regressions, reduces prepare + GC, purely algorithmic. Merge immediately.
2. Prepared scene cache — complementary to BPS. Cache reuse removes redundant prepare work that BPS makes cheaper but doesn’t eliminate.
3. Path caching — draw-time value unlocked after BPS + PSC reduce prepare. When prepare is no longer 95% of frame time, the 66% draw reduction becomes meaningful.
4. Spatial index — needs complete 24-row run before decision. Current partial results are modestly positive.
5. Native canvas — shelve. Path caching achieves a better draw reduction (66% vs 46%) without the prepare penalty. No path to standalone merge.

Methodology and Artifacts

How results were produced and where the raw data lives.

Comparison design

  • Baseline: all optimization flags false
  • Branch: only the single target flag set true
  • Same physical device (SM-F956B), same Android 34, same harness, same 24-scenario matrix
  • Self-tests validated in both runs
  • Matrix: 4 scene sizes × 2 mutation rates × 3 interaction patterns

Reading guide

  • avgFrameMs — primary decision metric (product-visible)
  • avgPrepareMs — subsystem diagnosis for scene prep
  • avgHitTestMs — subsystem diagnosis for interaction
  • allocatedMB / gcInvocations — cost-shifting indicators
  • p95 / p99 — tail latency when jank matters
