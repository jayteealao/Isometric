Performance Engineering on an Android Isometric Engine — Part 1: Measuring Before Optimising

Series overview. This is the first instalment of an ongoing record of performance work on the Isometric Kotlin library — a platform-agnostic 3D-to-2D isometric rendering engine with a Jetpack Compose runtime. Later parts will extend the story as new optimisations land. The goal is not a changelog but an explanation: why these choices, what the code is really doing, where the bodies are buried, and where the headroom still lives.

Why performance is harder than it looks for an isometric engine

An isometric engine is, on the surface, a simple thing. You take a list of 3D polygons, project them onto a 2D plane with a fixed oblique angle, sort them back-to-front, and paint them. There are no meshes, no shaders, no depth buffer. The whole pipeline is software, and it runs on the main thread on every frame.

That simplicity is both the engine's strength and its bottleneck. Because there is no GPU depth test, the engine has to determine draw order by reasoning about polygon geometry. Because there is no retained scene graph on the GPU, every frame that touches the scene must re-project every visible face from world space to screen space. And because the engine lives inside Jetpack Compose, the draw pipeline is entangled with the Compose recomposition lifecycle in ways that are easy to misunderstand.

The practical consequence is that frame time blows up faster than you expect as scene complexity grows. In the unoptimised baseline, a scene of 200 items renders in roughly 105–225 ms depending on interaction pattern — a range that spans from "barely acceptable on a 10fps display" to "visibly broken on any modern device". Understanding why it is that slow, and which part of the pipeline to attack first, required building a proper measurement framework before touching a single line of production code.

Part 1: Building the Benchmark Harness

The measurement problem with Compose

Measuring a Compose-based renderer is not the same as measuring a plain function. Compose's draw pipeline is asynchronous: Canvas.drawScope executes during the draw phase, which is scheduled by Choreographer and may happen one or more frames after the recomposition that produced the content. A naive approach — wrap IsometricScene in a LaunchedEffect, measure wall-clock time before and after, average over N frames — will produce measurements that mix recomposition time, layout time, draw time, and frame-pacing jitter in a way that makes it impossible to attribute cost to any specific subsystem.

The harness design document (written before a line of code was committed, which was intentional) identified three timing sites that needed to be measured independently:

  1. Prepare time — the time to run projectScene() inside IsometricEngine: transforming 3D points to 2D screen coordinates, applying lighting, running back-face culling, and sorting faces by depth. This is pure CPU computation with no Compose involvement.
  2. Draw time — the time to iterate the sorted PreparedScene.commands list and issue drawPath() calls on the Compose DrawScope. This is bounded by the number of visible faces and the cost of each Path object.
  3. Hit-test time — the time to identify which rendered shape is under a given screen coordinate. This matters because the benchmarks simulate user interaction (touch events), and interactive scenes have meaningfully different performance profiles from static ones.

These three times do not add up to the total frame budget. The difference is Compose overhead: recomposition, layout, the ForEach slot machinery, and the OS frame-pacing gap between Choreographer callbacks. The harness tracks that gap separately as frame time, measured as the interval between consecutive onDraw completions.

The LocalBenchmarkHooks contract

The key architectural decision was how to get timing data out of the renderer without contaminating the production code path. The chosen mechanism is a CompositionLocal:

// Production (in IsometricRenderer.draw):
benchmarkHooks?.onPrepareStart()
val scene = engine.projectScene(width, height, renderOptions, lightDirection)
benchmarkHooks?.onPrepareEnd()

LocalBenchmarkHooks defaults to null. In production builds running normally, the null-check short-circuits with no allocation and essentially zero overhead. In the benchmark Activity, a BenchmarkHooksImpl is provided via CompositionLocalProvider, and all timing calls flow through it to a MetricsCollector.

The forceRebuild flag on IsometricRenderer follows the same zero-overhead pattern: it is a property (not a function parameter) so that it can be set once before the benchmark loop without changing the render() / renderNative() call signatures. Setting it to true bypasses the prepared-scene cache on every frame, which is how the baseline measurement ensures it is measuring the real cost of projectScene() rather than a cache hit.

The 24-scenario matrix

The benchmark matrix was designed to span three independent dimensions:

DimensionValuesRationale
Scene size (N)10, 50, 100, 200Covers vsync-floor, sub-60fps, mid-range, and overloaded regimes
Mutation rate10%, 50%Low and high churn; affects cache hit rates and recomposition frequency
Interaction patternNONE, OCCASIONAL, CONTINUOUSDetermines hit-test load; CONTINUOUS fires every other frame

This gives 4 × 2 × 3 = 24 scenarios. Each scenario runs for 500 measurement frames across 3 iterations with an adaptive warmup phase (JIT needs time to stabilise; the harness waits until frame-time coefficient of variation drops below 10% over a 20-frame window before counting frames).

The "CONTINUOUS" interaction pattern deserves particular attention. It is not a realistic usage pattern — real users do not tap the screen 60 times a second. But it creates a useful stress test for the hit-test subsystem, and it has an interesting interaction with the prepared-scene cache that only reveals itself at scale (more on this in Part 3).

Self-tests as a correctness gate

Before the matrix runs, two self-tests execute:

  • selftest_cache — N=10, mutationRate=0, all caches ON. Asserts that the cache hit rate exceeds 95%. A static scene with no mutations should always hit the cache; if it doesn't, the cache is broken or (as was discovered) the engine reference is unstable.
  • selftest_sanity — N=10, mutationRate=0.10, all flags OFF. Asserts that average frame time is below 20ms and observed mutation rate is within ±2% of configured rate. This catches harness bugs (wrong mutation logic) and catastrophic regressions before they waste 10 minutes of benchmark time.

The self-tests proved their value almost immediately.

Part 2: The Prepared-Scene Cache — Skipping Work You Already Did

What projectScene() actually does

Every call to IsometricEngine.projectScene() does this, for every item in the scene:

val transformedItems = sceneGraph.items.mapNotNull { item ->
    projectAndCull(item, originX, originY, renderOptions, normalizedLight, width, height)
}

Inside projectAndCull, each face's polygon vertices are individually projected from 3D world space to 2D screen space using the isometric projection formula, back-face culled, bounds-checked against the viewport, and lit using the dot product of the face normal with the light direction. The result is a new List<Point2D> per face, a new IsoColor per face, and a new TransformedItem wrapping both. Then DepthSorter.sort() runs the O(N²) intersection test to produce the draw order.

For a 200-item scene with an average of 3 visible faces per item, this is roughly 600 projection calls, 600 lighting calculations, and a sort over 600 items — on every frame, even if the scene has not changed at all since last frame.

The cache design

The prepared-scene cache stores the last PreparedScene alongside the conditions under which it was produced. On the next call to projectScene(), it checks whether those conditions still hold:

fun needsUpdate(rootNode, context, width, height): Boolean {
    val currentInputs = PrepareInputs(context.renderOptions, context.lightDirection)
    return rootNode.isDirty ||
           !cacheValid ||
           width != cachedWidth ||
           height != cachedHeight ||
           currentInputs != cachedPrepareInputs ||
           engine.projectionVersion != cachedProjectionVersion
}

If none of these conditions is true, the cache is returned as-is. The only cost is six comparisons and a boolean OR — a few nanoseconds. The original commit message put it plainly: "fast path: 4 int comparisons + 1 reference check (~5ns). Slow path: full scene preparation (~65ms)."

The isDirty flag on GroupNode is the most important condition. It is set whenever any ShapeNode inside the scene tree changes — when a ForEach slot updates an item's position, colour, or geometry. The markClean() call at the end of a successful rebuild resets it. projectionVersion is a monotonically incrementing counter on IsometricEngine that only changes when the projection angle or scale changes at runtime; it signals that the 2D coordinates themselves would change even if the scene content hasn't.

The stable engine reference bug

The self-test immediately revealed a 0% cache hit rate — not because the cache logic was wrong, but because the engine it was caching on behalf of was being replaced on every recomposition.

The original BenchmarkScreen constructed AdvancedSceneConfig inline, like this:

IsometricScene(
    config = AdvancedSceneConfig(
        // engine defaults to IsometricEngine() — creates a NEW instance here
        renderOptions = RenderOptions.Default.copy(
            enableBroadPhaseSort = flags.enableBroadPhaseSort
        ),
        frameVersion = frameVersion,  // incremented every frame
        ...
    )
)

AdvancedSceneConfig.engine has a default parameter of IsometricEngine(). Every time BenchmarkScreen recomposes — which happens every time frameVersion increments, which happens every frame — AdvancedSceneConfig is instantiated with a fresh IsometricEngine() object.

Inside IsometricScene, the renderer is keyed on the engine reference:

val engine = remember(config.engine) { config.engine }
val renderer = remember(engine, ...) { IsometricRenderer(engine, ...) }

A new engine object means a new renderer object. A new renderer object means a fresh SceneCache with cacheValid = false. The very first condition in needsUpdate() — !cacheValid — fires on every single frame. The cache appeared to be working (the code was correct) but the object it lived in was being discarded before it could ever be reused.

The fix was one line:

val engine = remember { IsometricEngine() }

With the engine stabilised in remember{}, the renderer is created once and its cache accumulates state across frames. The self-test immediately passed with a 99% hit rate.

This is a good example of a class of Compose bugs where the contract is correct but the composition breaks it. The cache logic was fine; the problem was that Compose's object identity semantics require callers to ensure that the keys they pass to remember() are stable. A new IsometricEngine() is structurally identical to the previous one but referentially different — and reference equality is what remember() uses.

What the cache buys in practice

The cache hit rate at runtime is governed by probability. For a scene of N items where each item independently mutates with probability p per frame, the probability that a given frame is a cache hit (no item changed) is:

P(cache hit) = (1 - p)^N

At N=10, p=0.10: 0.9^10 ≈ 34.9% — consistent with the measured 32.4%.

At N=50, p=0.10: 0.9^50 ≈ 0.5% — consistent with the measured 0.2%.

At N=100 and above: effectively zero for any non-trivial mutation rate.

This means the prepared-scene cache only provides meaningful throughput improvement for small scenes or for the specific pattern where mutations are externally gated. The "CONTINUOUS" interaction pattern turns out to be the most valuable case: the orchestrator alternates between hit-test frames (no mutation) and mutation frames, which produces exactly a 50% cache hit rate regardless of scene size. This is what makes the cache genuinely useful at N=200 in the combined run — every other frame skips the entire projectScene() call.

The single-branch PSC benchmark confirmed this: PSC alone delivered a −17% average GC reduction and 2.40× speedup at N=200 — respectable, but not the headline number. The cache needs a companion optimisation to earn its keep for large scenes.

Alternative approaches considered

A version-counter approach (increment a scene version on every mutation, compare counters in the cache check) was the original design. The final implementation kept the isDirty flag on the node tree rather than a global version because it is cheaper to check (isDirty is set at the node that changed; a global counter requires an extra field and an atomic write per mutation). The downside is that isDirty is cleared by markClean() at the end of rebuild(), so it is not idempotent across multiple concurrent renders — but the renderer is single-threaded, so this is not a problem in practice.

A more aggressive approach would be per-item dirty tracking: cache the projected position and lit colour for each item individually, and only re-project the items that changed. This would make the cache useful at high mutation rates by only paying for the items that moved. The cost is significantly more complex bookkeeping, and it does not help with the sort phase — even if only one item changes, the sort must be rerun because that item's position relative to all others may have changed. Given that sort cost dominates (see Part 3), per-item cache refinement was deprioritised.

Part 3: Broad-Phase Sort — Taming the O(N²) Bottleneck

Why depth sorting is O(N²) by design

The isometric depth sort does not use a simple comparator. Isometric scenes can contain genuine depth ambiguities — two shapes that partially overlap on screen, where neither is entirely in front of the other — and a comparator-based sort does not handle these correctly or even consistently. The correct algorithm is a topological sort on a dependency graph: face A must be drawn before face B if A's projection lies behind B's from the viewer's perspective.

Building this graph requires testing every pair of faces for 2D screen-space overlap, then resolving the ordering using 3D depth. The naive implementation is O(N²):

for (i in 0 until length) {
    val itemA = items[i]
    for (j in 0 until i) {
        val itemB = items[j]
        if (IntersectionUtils.hasIntersection(...)) {
            val cmpPath = itemA.item.path.closerThan(itemB.item.path, observer)
            // record dependency
        }
    }
}

For a 200-item scene with 3 faces per item, this is 600 faces, roughly 180,000 pair tests per frame. Each test calls IntersectionUtils.hasIntersection(), which checks whether two convex polygons overlap in screen space. It is not a cheap operation.

The benchmarks confirmed the cost: at N=200, prepare time averaged 97–107 ms in the baseline. The sort was the reason.

The broad-phase grid

The fix is a classic technique from physics engines: before testing any pair, use a coarse spatial grid to eliminate pairs whose axis-aligned bounding boxes do not even overlap. Only pairs that share at least one grid cell need to be passed to the expensive intersection test.

private fun buildBroadPhaseCandidatePairs(
    items: List<TransformedItem>,
    cellSize: Double
): List<Pair<Int, Int>> {
    val grid = HashMap<Long, MutableList<Int>>()

    items.forEachIndexed { index, item ->
        val bounds = item.getBounds()
        val minCol = floor(bounds.minX / cellSize).toInt()
        val maxCol = floor(bounds.maxX / cellSize).toInt()
        val minRow = floor(bounds.minY / cellSize).toInt()
        val maxRow = floor(bounds.maxY / cellSize).toInt()

        for (row in minRow..maxRow) {
            for (col in minCol..maxCol) {
                grid.getOrPut(cellKey(col, row)) { mutableListOf() }.add(index)
            }
        }
    }
    // emit unique pairs from co-resident items
}

Each face is placed into every grid cell it overlaps. Then, for each cell, every pair of faces in that cell becomes a candidate for the expensive test. The cell key is packed into a Long as (col.toLong() shl 32) xor (row.toLong() and 0xffffffffL) — this avoids boxing and keeps the hash map fast. Pairs are deduplicated with a HashSet<Long> using the same bit-packing trick.

The performance impact was the largest single improvement measured: −55% GC, 3.60× speedup at N=200, −82% frame time for the worst-case s200_m50_continuous scenario. No other single optimisation came close.

Why it works so well for isometric scenes specifically

Isometric scenes have a structural property that makes broad-phase pruning particularly effective: faces tend to cluster spatially. A prism at position (x, y, z) projects to a contiguous patch of screen space. Its three visible faces all land near each other. A second prism at (x+5, y+5, z) projects to a completely different patch. The two prisms have no screen-space overlap, and the broad-phase grid rejects their face pairs without any intersection test at all.

The effective pruning ratio grows with scene density. In a sparse scene (N=10), most faces already have no neighbours in the same cell, and the broad phase adds overhead without saving much. This explains why BPS showed neutral results (±0%) for N=10 scenarios — the grid overhead matched the savings. At N=200, the pruning is dramatic: hundreds of far-apart faces are rejected instantly, and only geometrically proximate pairs are tested.

The cell size trade-off

The broadPhaseCellSize parameter (default: 100 pixels) determines the coarseness of the grid. Larger cells mean more false positives (pairs that share a cell but do not actually overlap), more expensive hasIntersection() calls, and less pruning. Smaller cells mean finer exclusion but more memory and more items in the grid map.

At the default scale (70 pixels per world unit), a typical face is 100–150 pixels wide on screen. A cell size of 100 pixels ensures that most faces span 1–2 cells, which is the sweet spot: cells are small enough to exclude distant pairs but large enough that each face does not span too many cells. The RenderOptions.broadPhaseCellSize is a public parameter, allowing callers who use non-default scales to tune this.

The require(broadPhaseCellSize > 0.0) guard on RenderOptions construction is a deliberate enforcement point — there is no meaningful interpretation of a zero or negative cell size, and failing early at the construction site is better than producing silently wrong output later.

What the tests check

The parity tests for broad-phase sort verify two things: that the sort produces identical draw order to the baseline for sparse scenes (where no pruning occurs), and for dense overlapping scenes (where the graph is non-trivial). This is important because the optimisation changes which pairs are tested but must not change which orderings are emitted. A bug in the deduplication logic that caused a pair to be tested twice would not change correctness (the dependency would just be recorded once anyway), but a bug that caused a pair to be missed would produce incorrect draw order — a painter's-algorithm violation that would manifest as a z-fighting flicker.

Part 4: Spatial Index for Hit Testing — From O(N) to O(k)

The hit-test bottleneck in interactive scenes

Hit testing answers the question: "which shape did the user touch?" The naive implementation iterates every RenderCommand in the prepared scene in reverse draw order (front-to-back) and tests whether the touch point lies inside the command's polygon. For a 200-item scene with 3 visible faces each, this is up to 600 polygon containment tests per touch event.

In NONE interaction (no touches), this cost is zero. In OCCASIONAL (a tap every few seconds), it is negligible. In CONTINUOUS (a simulated touch every other frame at 60fps), it contributes 102 ms to the frame at N=200 in the baseline — nearly the entire frame budget by itself.

The spatial index moves this from O(faces) to O(k) where k is the number of faces whose bounding boxes contain the touch point. For a sparse layout, k is typically 1–5; for a densely overlapping scene, k might be 10–20. In either case, the difference from 600 is dramatic.

A bug hidden in two layers

The spatial index implementation shipped with two bugs that cancelled each other out in basic testing but showed up in the benchmarks:

Bug 1: The index was only built inside the path-caching guard. The original code placed buildSpatialIndex() inside an if (enablePathCaching) block. Setting enableSpatialIndex=true without enablePathCaching=true would silently do nothing — the index was never built, and the "fast path" code would find spatialIndex == null and fall through to the slow path. The single-flag SI benchmark ran with path caching off, which means the entire spatial index branch measured the O(N) slow path and reported it as the spatial index result.

Bug 2: The "fast path" was O(k×N). The original code queried the spatial index for a list of candidate command IDs, then called engine.findItemAt() on the full prepared scene for each candidate. findItemAt() performs its own linear scan. The result was k linear scans instead of one — strictly worse than the original O(N) for k > 1.

The fix, committed as accc7a0, resolved both:

  1. Split buildCacheAndIndex() into independent buildPathCache() and buildSpatialIndex() methods, each guarded by their own flag.
  2. Changed the hit-test fast path to build a filtered PreparedScene from candidates and call findItemAt() once on that subset — achieving true O(k).
// Before (O(k*N)):
val candidates = spatialIndex!!.query(x, y)
for (commandId in candidates.asReversed()) {
    val filtered = PreparedScene(listOf(commandIdMap[commandId]!!), ...)
    val hit = engine.findItemAt(currentPreparedScene!!, x, y, ...)
    // searched the full scene for each candidate
}

// After (O(k)):
val candidates = spatialIndex!!.query(x, y)
val filteredScene = PreparedScene(
    candidates.mapNotNull { commandIdMap[it] },
    cachedWidth, cachedHeight
)
val hit = engine.findItemAt(filteredScene, x, y, ...)

The commit also introduced commandIdMap and commandToNodeMap — HashMap<String, RenderCommand> and HashMap<String, IsometricNode> respectively — to replace the find { } linear scans that were previously used to resolve command IDs to nodes after a hit test. These maps are rebuilt whenever the cache is rebuilt, which happens at most once per mutation cycle.

Why SI showed near-zero GC improvement

The spatial index branch added essentially no GC improvement (+7% vs baseline in the all-scenario average, within noise). The reason is structural: the SI optimisation targets hit-test time, not projectScene() time. The two largest allocation sites — the mapNotNull intermediate list in projectScene() and the sort comparator lambdas in DepthSorter.sort() — are untouched by SI. And the index itself allocates during every rebuild: the HashMap and bucket lists are reconstructed from scratch on each cache miss. For scenes that miss the cache on every frame (N≥50, mutationRate≥10%), the index rebuild overhead roughly equals whatever allocation savings come from faster hit testing.

SI's value is purely a latency win: it drives hit-test time from 102 ms to 30 ms at N=200 continuous, which matters enormously for interactivity but does not show up in GC numbers.

Part 5: Path Caching — The Draw Phase's Hidden Cost

What toComposePath() does on every frame

The draw phase iterates PreparedScene.commands and for each command calls drawPath() on the Compose DrawScope. Compose's DrawScope.drawPath() takes an androidx.compose.ui.graphics.Path object. The engine stores paths as its own io.github.jayteealao.isometric.Path type (a list of 3D Point objects). The conversion happens at draw time:

// Inside IsometricRenderer.draw(), for each command:
drawScope.drawPath(command.toComposePath(), color)

toComposePath() allocates a new android.graphics.Path object, moves to the first projected point, adds lines to the remaining points, and closes the polygon. For a 600-face N=200 scene, this is 600 Path allocations per draw call — roughly 3 allocations per face (the Path object, the internal PathData backing array, and the coordinate array).

This is not the most expensive thing the renderer does, but it is entirely unnecessary for frames where the scene geometry has not changed. The projected screen coordinates of a face are fixed until the cache is invalidated.

The path cache design

When enablePathCaching = true, the renderer maintains a List<SceneCache.CachedPath> that is rebuilt whenever SceneCache.rebuild() runs:

private fun buildPathCache(scene: PreparedScene) {
    cachedPaths = scene.commands.map { command ->
        CachedPath(
            path = command.toComposePath(),
            fillColor = command.color.toComposeColor(),
            commandId = command.commandId
        )
    }
}

On frames where the prepared-scene cache is valid, the draw loop uses the pre-converted paths directly. On frames where the cache is invalid (a mutation occurred), the path cache is rebuilt as part of the same rebuild operation. The net effect is that toComposePath() is called at most once per face per mutation event, not once per face per draw frame.

The PC benchmark showed this was primarily a draw-time and GC win, not a prepare-time win. Draw times dropped dramatically at all scene sizes — the path conversion was a meaningful fraction of draw cost. But GC showed nearly zero improvement at the all-scenario average (+2% vs baseline), because path allocation is small relative to the projectScene() intermediate lists that BPS and PSC eliminate. Path caching pays the most when the draw phase is the bottleneck, which only happens when the prepare phase has already been heavily optimised — exactly the position we are in with BPS active.

The ForEach/Compose recomposition allocations are the irreducible floor: every mutation causes a recomposition of the affected slots, which the Compose runtime handles through its slot table machinery. No engine-level optimisation can reduce this cost; it is the price of using Compose as the scene description layer.

Part 6: Native Canvas — Bypassing the Compose Drawing Abstraction

What the Compose DrawScope costs

When IsometricRenderer.draw() calls drawScope.drawPath(path, color), execution goes through several layers: DrawScope → DrawContext → Canvas → android.graphics.Canvas. Each layer adds indirection. The Paint object must be constructed and configured. The Compose graphics abstraction maintains internal state for blending, transform, and clipping that must be checked even when not used.

The native canvas rendering path bypasses this entirely by holding a reference to the underlying android.graphics.Canvas and calling drawPath() directly with a pre-configured Paint. The Compose fillPaint and strokePaint are now lazy properties:

private val fillPaint by lazy {
    Paint().apply {
        style = Paint.Style.FILL
        isAntiAlias = true
    }
}

The lazy was not just an optimisation; it was a correctness fix. Paint() calls into the Android graphics native library. Constructing a Paint at field initialisation time caused UnsatisfiedLinkError in JVM unit tests that did not load the Android native libraries. Making the construction lazy deferred it to the first actual draw call, which only happens on Android.

Why native canvas was left out of the combined run

The native canvas benchmark showed a small regression at N=10–100 (~4–8% slower) and a marginal improvement at N=200. This is counterintuitive but explainable: the Compose DrawScope abstraction has been heavily optimised by Google for exactly the use case of drawing many paths with solid fills. The drawPath overhead it adds is smaller than expected, and the Paint object reuse that native canvas enables is also something Compose does internally.

More importantly, native canvas is Android-specific. The isometric engine is designed to be platform-agnostic — the isometric-core module has no Android dependency, and the isometric-compose module's dependency on android.graphics.Canvas is already a compromise. Deepening that dependency for a marginal speedup that regresses half the workloads was not justified. Native canvas remains behind a flag for callers who need it for specific use cases (very large N=200+ scenes with dense overlap) and want to accept the Android-only constraint.

Part 7: The Combined Run — Non-Linear Combination Effects

What "all flags on" actually means

The combined run (20260319-173754-baseline-perf) enabled BPS, PSC, SI, and PC simultaneously with native canvas off. It ran on a Galaxy Z Fold 6 (SM-F956B, Android 14) — a flagship device with a 120Hz display, meaning the effective 60fps target is 8.33ms per frame.

The headline result across all 24 scenarios: average frame time dropped from 56.23 ms to 20.51 ms (−64%, 2.74× speedup). GC invocations dropped by the same proportion. The worst-case scenario — s200_m50_continuous, 200 items, 50% mutation rate, touch every other frame — went from 225.54 ms to 42.82 ms (−81%, 5.27× speedup).

17 of 24 scenarios moved into the "WIN" category (>10% improvement). The 7 neutral scenarios are all N=10, where frame time was already at the 8.34ms vsync floor in the baseline. There were zero regressions.

GC pressure at N=200: 3× reduction

At N=200, the GC invocation rate per frame dropped from 20.6 to 6.9 (−67%). The distribution of credit across optimisations:

  • BPS accounts for most of the reduction at all sizes — eliminating O(N²) sort comparator lambdas is the dominant allocation site.
  • PSC provides additional reduction proportional to the cache hit rate — on cache-hit frames, projectScene() does not run, so all its allocations are avoided.
  • SI and PC contribute near-zero GC improvement — they eliminate hit-test traversal and path conversion allocations, which are small relative to projectScene().

The allocated-MB numbers (per-iteration heap snapshots) show higher variance and should be interpreted with caution. A snapshot of live heap at the moment of measurement is sensitive to when the last GC cycle ran, not just to steady-state allocation rate. The GC invocation count — which counts how many times the runtime decided the allocation rate had exceeded its threshold — is the more stable signal.

Part 8: What Remains Open

The N=100 gap

The most important unresolved performance problem is N=100. The combined run achieves ~24ms per frame at N=100 — better than baseline by 2.24×, but still outside the 60fps window for interactive use. The primary cost is projectScene() on cache-miss frames: even with BPS, the sort over 300 faces (100 items × 3 faces) involves significant work.

The two most likely paths forward are:

Move the sort off the main thread. projectScene() is pure computation with no Compose dependency. It could run on a CoroutineDispatcher.Default thread while the Compose draw phase uses the previous frame's PreparedScene. This is a double-buffering approach: the scene preparation for frame N happens concurrently with the draw of frame N-1. The complexity cost is significant — you need synchronisation, a versioning protocol, and careful handling of the transition frame — but the latency benefit is clear.

GPU depth sorting. The docs/internal/plans/2026-03-18-webgpu-roadmap.md plan describes using WebGPU compute shaders for the depth sort. A GPU radix sort over projected depth values runs in microseconds for thousands of items. The depth ambiguity cases (where topological sort is required) are rare in practice — most faces have unambiguous ordering based on depth alone — and could be handled by a CPU fallback for the ambiguous subset. This is the Phase 1 objective of the WebGPU workstream.

The cache hit rate ceiling

The prepared-scene cache provides diminishing returns as mutation rate increases. At N=50, mutationRate=50%, the cache hit rate is 0% — every frame has at least one mutation. Improving cache utility for high-churn scenes would require per-item caching (re-project only the items that changed), which requires solving the sort problem for the changed items. This is a meaningful engineering investment and was deprioritised in favour of the broader algorithmic improvements.

The spatial index at scale

The spatial index currently shows near-zero benefit in the benchmarks because the hit-test scenarios simulate simple taps (point queries), and the O(N) brute-force scan is fast enough for the frame budgets we measured. For drag operations — where a hit test fires on every motion event — or for scenes with dense overlap where k is large, the spatial index would show clearer wins. The fix to the O(k×N) bug means the fast path is now genuinely O(k), and the deduplication work has been done; the gains simply did not materialise in this specific scenario matrix.

Native canvas revisited

The native canvas path underperformed expectations because Compose's DrawScope.drawPath is already heavily optimised for filled polygon drawing. The opportunity may be larger for stroke drawing or for scenes where the DrawScope batch-flush behaviour creates overhead. This was not tested in the current matrix. A targeted investigation with profiling (rather than benchmarking) might reveal specific draw call patterns where bypassing DrawScope is worth the API compromise.

Reflection: What the Process Revealed

Measure first, always

The cache bug found by the self-test is the clearest argument for building a measurement harness before writing a single optimisation. Without the 0% cache hit rate, the PSC code would have appeared to run — no crash, no obviously wrong output — and the benchmarks would have been measuring BenchmarkScreen's recomposition cost instead of the renderer's cache behaviour. The self-test turned a latent correctness bug into an immediate, actionable failure.

The O(N²) problem was the right first target

BPS's dominance in both frame-time improvement (−66% at N=200) and GC reduction (−55% at all sizes) confirms that the sort was the correct first target. This is a general principle: when a pipeline has a stage that is algorithmically super-linear, fix the algorithm before tuning the implementation. All the constant-factor improvements (path caching, native canvas) contribute modest gains on top of a fast sort, but would have been invisible on top of an O(N²) bottleneck.

Single-flag isolation revealed unexpected interactions

Running each flag independently before combining them revealed that SI and PC, despite being sound optimisations, contribute nearly nothing to GC metrics. This was not obvious before measurement. SI and PC became visibly valuable only after BPS reduced the preparation cost to a level where draw-phase and hit-test costs began to dominate the frame budget. This is a general pattern in layered performance work: early optimisations reveal the shape of the next problem.
