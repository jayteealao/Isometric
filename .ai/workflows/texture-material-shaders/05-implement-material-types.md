# Implement: Material Type System & Module

## Summary of Changes

Created the `isometric-shader` Android library module with the complete material type
system (`IsometricMaterial` sealed interface, `TextureSource`, `UvCoord`, `UvTransform`),
DSL builders (`flatColor()`, `textured()`, `texturedAsset()`, `texturedBitmap()`,
`perFace()`), and wired the `material` parameter through `RenderCommand`, `ShapeNode`,
`PathNode`, and the `Shape()`/`Path()` composables. All changes are backward compatible —
existing `Shape(geometry, color)` call sites compile and render identically.

## Shared Files (also touched by sibling slices)
- `RenderCommand.kt` — will be consumed by `uv-generation` (writes `uvCoords`), `canvas-textures` (reads `material`), `webgpu-textures` (reads `material`)
- `IsometricNode.kt` — will be consumed by `per-face-materials` (resolves `PerFace` per path)
- `IsometricComposables.kt` — stable after this slice; sibling slices don't modify composable signatures

## Notes on Design Choices
- `MaterialData` marker interface in `isometric-core` keeps the pure-JVM module free of Android dependencies while providing compile-time safety (vs `Any?`)
- `UvCoord` has no range validation — tiling requires values > 1.0, and the shader's TileMode handles out-of-range values
- `TextureSource.BitmapSource` (not `Bitmap`) to avoid shadowing `android.graphics.Bitmap`
- `@IsometricMaterialDsl` prevents scope leakage in nested `perFace { textured { ... } }` builders
- `material` parameter placed after `color` and before `alpha` in composable signatures for progressive disclosure

## Known Risks / Caveats
- Adding a new `IsometricMaterial` subtype is a breaking change for exhaustive `when` consumers (documented in KDoc)
- `TextureSource.BitmapSource` init guard only checks `isRecycled` at construction — bitmap could be recycled later
